01 What is dynamic programming?

Dynamic programming is an optimization technique that solves complex problems by breaking them into overlapping subproblems, solving each exactly once, and storing the results. It's applicable when two properties hold:

Optimal substructure

The optimal solution to the whole can be built from optimal solutions to subproblems.

Overlapping subproblems

The same subproblems recur many times — unlike divide-and-conquer, which has distinct subproblems.

mental model
Think of it as recursion + caching. If the naïve recursive solution recomputes the same call multiple times, you have overlapping subproblems — and DP is your tool.

02 When does DP apply?

Look for these signals in the problem statement:

  • "find the minimum / maximum number of ways…"
  • "how many distinct ways…"
  • "can you partition / achieve exactly…"
  • The input is a string, array, or grid and you're asked about subsequences or subarrays
  • The problem involves making a sequence of choices where each choice affects future options
don't confuse with
Greedy (one local choice is always globally optimal) and divide-and-conquer (subproblems don't overlap). If the subproblems are independent, plain recursion or merge sort is enough.

04 Approach 1: Memoization (top-down)

Write the natural recursion, then cache results in a dictionary. Execution flows top-down: you call the big problem, which recurses into subproblems on demand.

when to prefer it
Not all subproblems may be needed. Memoization naturally skips them. Also easier to reason about when the state space is sparse or the recurrence isn't obvious upfront.

Template

python
from functools import cache

def solve(problem_input):
    # @cache turns the function into a memoized version
    @cache
    def dp(state):
        if base_case(state):
            return base_value
        return combine(dp(substate1), dp(substate2), ...)

    return dp(initial_state)

@cache (Python 3.9+) or @lru_cache(None) — both memoize automatically. For mutable inputs (lists), convert to tuples before passing as the state.

05 Approach 2: Tabulation (bottom-up)

Fill a table from the smallest subproblems upward. No recursion stack — often faster in practice and easier to optimize memory.

when to prefer it
You need tight memory control (rolling array optimization), or the recurrence is clear and all subproblems will be visited anyway.

Template


def solve(n):
    dp = [0] * (n + 1)
    dp[0] = base_case_value          # seed the base case(s)

    for i in range(1, n + 1):
        dp[i] = recurrence(dp, i)   # fill left-to-right

    return dp[n]

06 Reconstructing the solution

The DP table gives you the value of the optimal solution. To get the actual path or sequence, trace backwards through the table using the same recurrence, choosing the transition that was taken. 


# After filling dp[], walk back to find the choices made
def reconstruct(dp, choices_or_parent):
    path = []
    i = len(dp) - 1
    while i > 0:
        choice = choices_or_parent[i]
        path.append(choice)
        i -= choice                  # step back by what was chosen
    return path[::-1]               # reverse to get forward order

P1 Fibonacci (warm-up)

The "hello world" of DP. Naïve recursion is O(2ⁿ); memoization or tabulation drops it to O(n).

Memoized


from functools import cache

@cache
def fib(n):
    if n <= 1: return n
    return fib(n - 1) + fib(n - 2)

Tabulated (O(1) space with rolling vars)


def fib(n):
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a
execution trace · fib(5) fib(5) → fib(4) + fib(3)
  fib(4) → fib(3) + fib(2)   ← without cache: fib(3) recomputed 2×
    fib(3) → fib(2) + fib(1)
      fib(2) → fib(1) + fib(0) = 1
    fib(3) = 2  (cached → instant)
  fib(4) = 3
fib(5) = 5

P2 0/1 Knapsack

Given n items each with weight and value, and a bag of capacity W, maximize total value without exceeding weight. Each item can be taken at most once (0/1).

recurrence
dp[i][w] = max(dp[i-1][w], dp[i-1][w - wt[i]] + val[i])
Either skip item i (take row above), or include it (go back by its weight, add its value).

Bottom-up, space-optimized to 1D


def knapsack(weights, values, W):
    n = len(weights)
    dp = [0] * (W + 1)

    for i in range(n):
        # Iterate W → weight[i] to avoid using item twice
        for w in range(W, weights[i] - 1, -1):
            dp[w] = max(dp[w], dp[w - weights[i]] + values[i])

    return dp[W]

# Example
weights = [2, 3, 4, 5]
values  = [3, 4, 5, 6]
print(knapsack(weights, values, W=8))  # → 10
critical detail
Iterate the capacity backwards in the 1D version. If you go forwards, an item can be picked multiple times (that's the unbounded knapsack variant — a different problem).

P3 Longest Common Subsequence

Find the longest subsequence common to two strings s and t (characters don't need to be contiguous, just in order).

recurrence
if s[i] == t[j]: dp[i][j] = dp[i-1][j-1] + 1
else: dp[i][j] = max(dp[i-1][j], dp[i][j-1]) 

def lcs(s, t):
    m, n = len(s), len(t)
    dp = [[0] * (n + 1) for _ in range(m + 1)]

    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if s[i-1] == t[j-1]:
                dp[i][j] = dp[i-1][j-1] + 1
            else:
                dp[i][j] = max(dp[i-1][j], dp[i][j-1])

    return dp[m][n]

print(lcs("ABCBDAB", "BDCABA"))  # → 4 ("BCBA")
cousin problems
Edit distance (Levenshtein), shortest common supersequence, longest palindromic subsequence — all share this 2D string DP structure with small recurrence tweaks.

P4 Coin Change

Given coin denominations and a target amount, find the minimum number of coins that sum to the target. Coins can be reused (unbounded).

recurrence
dp[a] = min(dp[a - coin] + 1) for each coin ≤ a 

def coin_change(coins, amount):
    INF = float('inf')
    dp = [INF] * (amount + 1)
    dp[0] = 0                      # 0 coins needed for amount 0

    for a in range(1, amount + 1):
        for coin in coins:
            if coin <= a:
                dp[a] = min(dp[a], dp[a - coin] + 1)

    return dp[amount] if dp[amount] != INF else -1

print(coin_change([1, 5, 6, 9], 11))  # → 2  (5+6)
table trace · coins=[1,5,6,9], amount=11 dp[0] = 0
dp[1] = 1 (0+1)
dp[5] = 1 (0+5)
dp[6] = 1 (0+6)
dp[10] = 2 (dp[5]+5 or dp[4]+6)
dp[11] = 2 (dp[5]+6 = 1+1)
count variant
Swap min for += and initialize dp[0]=1 to count the number of ways to make the amount — same structure, different aggregator.

P5 Longest Increasing Subsequence

Find the length of the longest strictly increasing subsequence in an array. The O(n²) DP solution is easy to derive; O(n log n) uses binary search.

O(n²) — clean DP


def lis(nums):
    n = len(nums)
    dp = [1] * n            # every element is an LIS of length 1

    for i in range(1, n):
        for j in range(i):
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)

    return max(dp)

print(lis([10,9,2,5,3,7,101,18]))  # → 4 (2,3,7,18)

O(n log n) — patience sorting


import bisect

def lis_fast(nums):
    tails = []                        # tails[i] = smallest tail of IS of len i+1
    for x in nums:
        pos = bisect.bisect_left(tails, x)
        if pos == len(tails):
            tails.append(x)           # extend
        else:
            tails[pos] = x            # replace with smaller tail
    return len(tails)

P6 Grid paths

Count unique paths from top-left to bottom-right of an m×n grid, moving only right or down. With obstacles, skip blocked cells.

recurrence
dp[i][j] = dp[i-1][j] + dp[i][j-1]
Paths to any cell = paths coming from above + paths coming from left. 

def unique_paths(m, n, obstacles=None):
    obs = obstacles or set()
    dp = [[0] * n for _ in range(m)]
    dp[0][0] = 1

    for i in range(m):
        for j in range(n):
            if (i, j) in obs or (i == 0 and j == 0): continue
            from_top  = dp[i-1][j] if i > 0 else 0
            from_left = dp[i][j-1] if j > 0 else 0
            dp[i][j] = from_top + from_left

    return dp[m-1][n-1]

print(unique_paths(3, 7))  # → 28

07 Interview framework

When you see a DP problem in an interview, work through this sequence out loud:

1 — Identify Is this DP? Look for "min/max", "how many ways", "can we achieve X".
2 — Define state What is dp[i] or dp[i][j]? Say it in plain English: "dp[i] = minimum coins to make amount i."
3 — Write recurrence Express dp[i] in terms of smaller subproblems. Cover all cases.
4 — Base cases What's the smallest valid state? Seed it explicitly.
5 — Order of evaluation Ensure each dp[i] is computed only after the states it depends on.
6 — Answer Where in the table is the final answer? It's not always dp[n].
7 — Optimize Can you reduce from 2D to 1D? Can you drop rows you no longer need?

08 Before you submit — checklist

  • State definition stated explicitly in words
  • Recurrence covers all transitions (including "skip" / "don't take")
  • Base case(s) initialized correctly (dp[0], dp[0][0], etc.)
  • Table filled in correct dependency order
  • Answer extracted from correct cell
  • Edge cases: empty input, n=0, amount=0, single element
  • Time complexity: O(states × transitions per state)
  • Space complexity noted; 1D optimization considered
  • Reconstruction logic if the actual path/choices are needed
  • Overflow / infinity initialization correct (float('inf'), not -1 as ∞)

The hardest part of DP is always the state definition. Once you have that right, the recurrence almost writes itself. When stuck, ask: "what information do I need to make the optimal choice at step i?" — that's your state.
