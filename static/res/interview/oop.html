Objects are
contracts, not containers

Python's OOP is pragmatic: everything is an object, duck typing beats strict hierarchies, and the data model lets you hook into the language itself via magic methods. Know the four pillars cold — then know Python's quirks on top of them.

Python 3.10+ 4 pillars Magic methods MRO Dataclasses Protocols SOLID

01 Four pillars

Encapsulation

Bundle data + behavior in a class. Hide internals; expose a clean interface. Controls who can see and mutate state.

Inheritance

A child class reuses and extends a parent. Python supports multiple inheritance; MRO determines resolution order.

Polymorphism

The same interface works on different types. Python achieves this via duck typing — if it quacks, it's a duck.

Abstraction

Expose what's needed; hide complexity. Formalized with ABCs or Protocols — callers depend on the interface, not the implementation.

02 Classes & instances

class BankAccount:
    interest_rate = 0.03          # class variable — shared across all instances

    def __init__(self, owner: str, balance: float = 0):
        self.owner   = owner       # instance variable — unique per object
        self._balance = balance    # _single underscore = "internal, please don't touch"
        self.__id = id(self)       # __double = name-mangled to _BankAccount__id

    def deposit(self, amount: float) -> None:
        if amount <= 0:
            raise ValueError("Deposit must be positive")
        self._balance += amount

    def __repr__(self) -> str:
        return f"BankAccount({self.owner!r}, balance={self._balance})"
Python convention on visibility
Python has no true private. _x is a hint ("don't use outside the class"). __x triggers name mangling to _ClassName__x — it's not private, just harder to access accidentally. There's no private keyword. Discipline is by convention.

03 Method types

class Circle:
    PI = 3.14159

    def __init__(self, radius):
        self.radius = radius

    def area(self):                    # instance method — receives self
        return Circle.PI * self.radius ** 2

    @classmethod
    def unit(cls):                     # class method — receives cls, not self
        return cls(1)                  # useful as an alternative constructor

    @staticmethod
    def validate_radius(r):           # static — no self, no cls; just a namespaced function
        return r > 0

# Usage
c1 = Circle(5)
c2 = Circle.unit()                # alternative constructor via classmethod
Circle.validate_radius(3)        # called on the class itself
when to use each
Use instance methods for anything that needs object state. Use classmethods for factory/alternative constructors (e.g., date.fromisoformat). Use staticmethods for utility logic that logically belongs to the class but needs no state.

04 Magic methods (dunder)

These let your objects participate in Python's built-in protocols — operators, iteration, context managers, and more.

__repr__
Dev-facing string: repr(obj)
__str__
User-facing string: str(obj), print()

Equality and dict/set membership. Define both together.
__len__
Called by len(obj)

Enables obj[key] syntax

Makes object iterable in for loops

Context manager: with obj:
__add__ / __mul__
Operator overloading: +, *, etc.
__call__
Makes instance callable: obj()
__contains__
Powers in operator
__del__
Destructor — called when GC collects the object

Practical example — a Vector class

class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

    def __add__(self, other):              # v1 + v2
        return Vector(self.x + other.x, self.y + other.y)

    def __mul__(self, scalar):             # v * 3
        return Vector(self.x * scalar, self.y * scalar)

    def __abs__(self):                     # abs(v) → magnitude
        return (self.x**2 + self.y**2) ** 0.5

    def __eq__(self, other):
        return isinstance(other, Vector) and self.x == other.x and self.y == other.y

Vector(1, 2) + Vector(3, 4)   # → Vector(4, 6)
abs(Vector(3, 4))             # → 5.0

05 Encapsulation

Encapsulation means an object owns its state and decides what the outside world can do with it. In Python this is enforced by convention and, more powerfully, by properties. 

class Temperature:
    def __init__(self, celsius: float):
        self.celsius = celsius       # uses the setter below

    @property
    def celsius(self) -> float:
        return self._celsius

    @celsius.setter
    def celsius(self, value: float):
        if value < -273.15:
            raise ValueError("Below absolute zero")
        self._celsius = value

    @property
    def fahrenheit(self) -> float:   # derived, read-only property
        return self._celsius * 9/5 + 32

t = Temperature(100)
print(t.fahrenheit)              # 212.0 — looks like an attribute, runs code
t.celsius = -300               # → ValueError

06 Inheritance & MRO

A child class inherits all methods and attributes of its parent. Use super() to call the parent's implementation. With multiple inheritance, Python uses the C3 linearization algorithm to determine Method Resolution Order (MRO). 

class Animal:
    def __init__(self, name):
        self.name = name

    def speak(self) -> str:
        raise NotImplementedError

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)       # delegates to Animal.__init__
        self.breed = breed

    def speak(self) -> str:
        return f"{self.name} says: Woof!"

# Multiple inheritance — MRO matters
class Flyable:
    def fly(self): return "I can fly"

class FlyingDog(Dog, Flyable):   # MRO: FlyingDog → Dog → Animal → Flyable → object
    pass

print(FlyingDog.__mro__)          # inspect the resolution chain
diamond problem
When two parents share a common ancestor, the MRO ensures the ancestor's method runs only once. Python's C3 algorithm guarantees a consistent, predictable order — always check __mro__ when debugging multiple inheritance surprises.
MRO trace · FlyingDog FlyingDog → Dog → Animal → Flyable → object
✓ speak() found on Dog — stops here, doesn't continue
✓ fly() not on Dog or Animal — found on Flyable
✓ __init__ found on Dog — calls super() → Animal.__init__

07 Polymorphism

Different objects responding to the same interface. Python's approach is duck typing: if an object has the right methods, it works — no shared base class required. 

class Cat:
    def speak(self): return "Meow"

class Dog:
    def speak(self): return "Woof"

class Robot:
    def speak(self): return "Beep boop"

def make_noise(things):          # no type constraint — any object with .speak() works
    for t in things:
        print(t.speak())

make_noise([Cat(), Dog(), Robot()])
# Meow / Woof / Beep boop
duck typing in practice
You almost never need isinstance() checks in well-designed Python code. If you find yourself writing if isinstance(x, Dog), consider whether a protocol or overloaded method would be cleaner.

08 Abstraction

Define an interface that subclasses must implement. Python offers two mechanisms: ABCs (enforced at instantiation) and Protocols (structural, checked by type checkers).

Abstract Base Class (ABC)

from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self) -> float: ...

    @abstractmethod
    def perimeter(self) -> float: ...

class Rectangle(Shape):
    def __init__(self, w, h):
        self.w, self.h = w, h

    def area(self):      return self.w * self.h
    def perimeter(self): return 2 * (self.w + self.h)

# Shape()  → TypeError: Can't instantiate abstract class
# Rectangle(3, 4) → works fine

09 Properties in depth

class Person:
    def __init__(self, name):
        self._name = name

    @property
    def name(self):        # getter
        return self._name

    @name.setter
    def name(self, value): # setter
        if not isinstance(value, str):
            raise TypeError("Name must be a string")
        self._name = value.strip()

    @name.deleter
    def name(self):        # deleter — del obj.name
        self._name = "Anonymous"
key insight
Properties let you start with a plain attribute and add validation later without changing the public API. Callers still write person.name = "Alice" — they never know there's a setter underneath.

10 Dataclasses

Auto-generates __init__, __repr__, __eq__ — and more with flags.

from dataclasses import dataclass, field

@dataclass
class Point:
    x: float
    y: float
    label: str = "unnamed"         # fields with defaults come last

@dataclass(frozen=True)            # immutable — instances are hashable
class ImmutablePoint:
    x: float
    y: float

@dataclass(order=True)             # generates __lt__, __le__, etc.
class Player:
    score: int
    name: str
    tags: list = field(default_factory=list)  # never use [] as default!
common trap
Never use a mutable default like tags: list = [] — the same list object is shared across all instances. Always use field(default_factory=list).

11 Protocols (structural subtyping)

A Protocol defines an interface purely by shape — no inheritance required. Any class with the right methods satisfies it. Type checkers enforce it; Python's runtime doesn't. 

from typing import Protocol

class Drawable(Protocol):
    def draw(self) -> None: ...

class Circle:                       # does NOT inherit from Drawable
    def draw(self): print("⬤")

def render(shape: Drawable) -> None:
    shape.draw()

render(Circle())  # ✓ — Circle satisfies Drawable structurally
ABC vs Protocol
Use ABC when you want runtime enforcement and want to share implementation (mixin methods). Use Protocol when you want zero coupling — callers define what they need, implementors don't know about it.

12 __slots__

By default, every Python object has a __dict__ — a hash map of its attributes. Declaring __slots__ replaces it with a fixed-layout array: faster access, less memory. 

class Point:
    __slots__ = ('x', 'y')          # only x and y allowed — no __dict__

    def __init__(self, x, y):
        self.x, self.y = x, y

p = Point(1, 2)
p.z = 3                            # → AttributeError: 'Point' has no attribute 'z'
when to use it
Use __slots__ for objects created in the millions (e.g., nodes in a graph or tree). Typically saves ~40–50% memory per instance. Don't bother for normal application-level objects.

13 Metaclasses (know the concept)

A metaclass is the class of a class. type is the default metaclass of every Python class. Metaclasses let you intercept class creation — not just instance creation. 

class SingletonMeta(type):
    _instances = {}

    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

class Config(metaclass=SingletonMeta):
    def __init__(self): self.debug = False

Config() is Config()              # → True — always the same object
interview note
You rarely write metaclasses in application code. Know they exist, know what they do, and know that __init_subclass__ and class decorators handle 80% of metaclass use cases more cleanly.

14 SOLID + Python specifics

PrincipleAbbrev.In plain Python
Single ResponsibilitySOne class, one reason to change. A User class shouldn't also send emails.
Open/ClosedOOpen for extension (subclass or inject), closed for modification. Use ABCs / Protocols.
Liskov SubstitutionLA subclass must be usable wherever the parent is. Don't override to raise exceptions the parent wouldn't.
Interface SegregationIMany small Protocols beat one fat ABC. Don't force a class to implement methods it doesn't need.
Dependency InversionDDepend on abstractions (Protocol / ABC), not concrete classes. Pass dependencies in; don't import them inside.
composition over inheritance
Favor composing behavior (passing objects in) over deep inheritance chains. Deep hierarchies are fragile — a change at the top breaks everything below. In Python, mixins and Protocols usually get you further than 5-level inheritance.

15 Before you interview — checklist

  • Can name and explain the four pillars with a concrete Python example each
  • Knows the difference between _x, __x, and x
  • Can write a @property with getter, setter, and deleter from memory
  • Understands @classmethod vs @staticmethod vs instance method
  • Can explain MRO and predict output of a diamond inheritance scenario
  • Knows when to use ABC vs Protocol — and the trade-offs
  • Understands __eq__ and __hash__ must be defined together
  • Can implement a context manager via __enter__ / __exit__
  • Knows the mutable default trap in dataclasses (and regular __init__)
  • Can explain Liskov Substitution with an example of a violation
  • Knows __slots__ purpose and when it matters
  • Can sketch the Singleton, Factory, and Observer patterns in Python

The hardest OOP interview questions aren't about syntax — they're about design decisions. When to use inheritance vs composition, when ABC vs Protocol, why Liskov matters. Know the why behind each concept, and the code will follow.
