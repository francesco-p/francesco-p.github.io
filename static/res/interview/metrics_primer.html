Interview Primer · ML Metrics

What you measure
is what you optimize

Every metric encodes a decision about what errors cost. Knowing the formula is table stakes — the interview tests whether you know when each metric lies, what its failure modes are, and how to choose between them when they disagree.

01 Accuracy
Classification
formula Accuracy = (TP + TN) / (TP + TN + FP + FN)
= correctly classified / total samples
Range: [0, 1]. Higher is better.

What it measures: the fraction of all predictions that were correct, across all classes.

python

from sklearn.metrics import accuracy_score
import numpy as np

y_true = [1, 0, 1, 1, 0]
y_pred = [1, 0, 0, 1, 1]  # 3 correct out of 5

accuracy_score(y_true, y_pred)    # → 0.6

# Manual
np.mean(np.array(y_true) == np.array(y_pred))  # → 0.6
the class imbalance trap
With 99% negative samples, predicting "always negative" gives 99% accuracy — and catches zero positives. Accuracy is meaningless when classes are imbalanced. Always ask: what's the majority class baseline? If accuracy ≈ majority class frequency, the model learned nothing.
✓ use when

Classes are roughly balanced. Random chance baseline is meaningful. Cost of FP ≈ cost of FN.

✗ avoid when

Imbalanced data. One type of error is more costly (fraud, disease, defect detection). Multi-class with skewed class distributions.

02 Precision
Classification
formula Precision = TP / (TP + FP)
= of everything predicted positive, how many actually were
Range: [0, 1]. A model predicting nothing positive has undefined (0/0) precision.

What it measures: the quality of positive predictions — when you raise the alarm, how often are you right?

Optimize precision when false positives are costly: spam filters (legitimate email wrongly blocked), surgery recommendations (operating on a healthy patient), drug approvals (side effects from unnecessary treatment).

python

from sklearn.metrics import precision_score

# Binary
precision_score(y_true, y_pred)
# → TP / (TP + FP)

# Multi-class: average across classes
precision_score(y_true, y_pred, average='macro')    # unweighted mean
precision_score(y_true, y_pred, average='weighted') # weighted by class support
precision_score(y_true, y_pred, average='micro')    # globally over all samples
precision-recall tradeoff
Raising your classification threshold increases precision (you only predict positive when very confident) but decreases recall (you miss more true positives). They move in opposite directions. There is no free lunch — you must choose based on what errors cost more in your domain.
03 Recall (Sensitivity / TPR)
Classification
formula Recall = TP / (TP + FN)
= of all actual positives, how many did we catch

Also called: Sensitivity, True Positive Rate (TPR), Hit Rate
Range: [0, 1]. A model predicting everything positive has recall = 1.0 trivially.

What it measures: completeness — how many real positives do you catch? Optimizing recall means minimizing missed detections.

Optimize recall when false negatives are costly: cancer screening (missing a diagnosis), fraud detection (letting a fraud transaction through), safety systems (missing a pedestrian), defect detection (a defective part reaches the customer).

python

from sklearn.metrics import recall_score

recall_score(y_true, y_pred)  # → TP / (TP + FN)

# Specificity (True Negative Rate) — often confused with recall
# Specificity = TN / (TN + FP)  — how well we identify negatives
# Recall = TPR. Specificity = TNR. Together they define the ROC curve.
trivial high recall
A model that predicts everything as positive achieves recall = 1.0 (catches all true positives) but precision collapses. This is why you never evaluate recall in isolation — always with precision or as F1/AUC-PR.
04 F1 Score
Classification
formula F1 = 2 · (Precision · Recall) / (Precision + Recall)
= harmonic mean of precision and recall

generalized Fβ Fβ = (1 + β²) · (P · R) / (β²·P + R)
β > 1 → weights recall more    β < 1 → weights precision more
F1 = F_β with β=1. F2 gives recall twice the weight of precision.

Why harmonic mean, not arithmetic mean? The harmonic mean is dominated by the smaller value. If precision = 1.0 and recall = 0.01, arithmetic mean = 0.505 (misleadingly good); harmonic mean = 0.02 (reflects the poor recall). A metric that rewards gaming either one alone is useless.

python

from sklearn.metrics import f1_score, fbeta_score

f1_score(y_true, y_pred)                        # binary F1
f1_score(y_true, y_pred, average='macro')       # unweighted mean over classes
f1_score(y_true, y_pred, average='weighted')   # weighted by class frequency

fbeta_score(y_true, y_pred, beta=2)            # F2: recall-heavy
fbeta_score(y_true, y_pred, beta=0.5)          # F0.5: precision-heavy

# Manual:
p = precision_score(y_true, y_pred)
r = recall_score(y_true, y_pred)
f1 = 2 * p * r / (p + r)

macro vs micro vs weighted averaging

AverageHowWhen
macroCompute F1 per class, take unweighted meanWhen all classes matter equally, regardless of size. Penalizes poor performance on rare classes.
weightedCompute F1 per class, weight by class supportWhen class imbalance exists but large classes are more important. Common in practice.
microAggregate TP/FP/FN globally, then computeWhen you care about per-sample performance equally. Dominated by large classes.
F1 doesn't account for true negatives
F1 ignores TN — it only cares about positive predictions. For highly imbalanced data where the negative class dominates, F1 can give a misleading picture. Matthews Correlation Coefficient (MCC) is a more balanced alternative that includes all four quadrants: MCC = (TP·TN − FP·FN) / √((TP+FP)(TP+FN)(TN+FP)(TN+FN)).
05 AUC-ROC
Ranking · Threshold-free
ROC curve x-axis: FPR = FP / (FP + TN)   (1 − Specificity)
y-axis: TPR = TP / (TP + FN)   (= Recall)
Sweep threshold from 1 → 0, plot (FPR, TPR) at each step

AUC (area under curve) AUC = 0.5 → random classifier
AUC = 1.0 → perfect ranking
AUC = P(score(pos) > score(neg))   ← probabilistic interpretation

What AUC-ROC actually measures: given a randomly drawn positive and negative sample, what's the probability your model assigns a higher score to the positive? It's a measure of ranking quality, not classification quality.

AUC-ROC is threshold-independent: you don't need to choose a classification threshold to compute it. This makes it useful for model comparison regardless of operating point.

python

from sklearn.metrics import roc_auc_score, roc_curve
import matplotlib.pyplot as plt

y_scores = model.predict_proba(X_test)[:, 1]  # positive class proba

auc = roc_auc_score(y_true, y_scores)

fpr, tpr, thresholds = roc_curve(y_true, y_scores)
plt.plot(fpr, tpr, label=f"AUC = {auc:.3f}")
plt.plot([0,1], [0,1], '--', color='grey')  # random baseline

# Multi-class AUC (one-vs-rest)
roc_auc_score(y_true_mc, y_scores_mc, multi_class='ovr', average='macro')
AUC-ROC lies under heavy imbalance
With 1000 negatives and 10 positives, even a bad model can achieve AUC ≈ 0.9 by correctly ranking most negatives below most positives — but it may still miss 80% of actual positives. The ROC curve is insensitive to the imbalance ratio because FPR is normalized by total negatives. Under heavy imbalance, AUC-PR is the correct metric.
06 AUC-PR (Precision-Recall Curve)
Ranking · Imbalance-aware
PR curve x-axis: Recall = TP / (TP + FN)
y-axis: Precision = TP / (TP + FP)
Sweep threshold 1 → 0, plot (Recall, Precision) at each step

Baseline = class prevalence = positives / total
Under 1% prevalence, random classifier AUC-PR ≈ 0.01 — not 0.5 like ROC

Why AUC-PR is better than AUC-ROC under imbalance: The PR curve uses precision on the y-axis, which directly penalizes false positives relative to true positives. It doesn't dilute the signal with the large number of true negatives that inflate AUC-ROC under imbalance.

Average Precision (AP) is the area under the PR curve computed as a weighted mean of precisions at each recall threshold — equivalent to AUC-PR for binary classification and identical to what mAP uses in object detection.

python

from sklearn.metrics import (
    average_precision_score, precision_recall_curve
)

ap = average_precision_score(y_true, y_scores)
# → scalar, equivalent to AUC of PR curve

prec, rec, thresh = precision_recall_curve(y_true, y_scores)
# prec[-1] = 1.0, rec[-1] = 0.0 (convention)

# Practical: always compare AUC-PR against the no-skill baseline
baseline = y_true.mean()   # prevalence — what a random model gets
print(f"AP={ap:.3f}, baseline={baseline:.3f}, lift={ap/baseline:.1f}x")
AP vs AUC-ROC — when they disagree
If AUC-ROC is high but AP is low, your model ranks well in aggregate but fails to produce precise predictions at the operating points that matter (high recall). This happens in medical screening or fraud detection — AUC-ROC looks fine but you can't set a threshold that gives acceptable precision without sacrificing too much recall.
07 MAE — Mean Absolute Error
Regression
formula MAE = (1/n) · Σ |yᵢ − ŷᵢ|

Unit: same as the target variable
Range: [0, ∞). Corresponds to optimizing L1 loss. Equivalent to Laplace distribution likelihood.

What it measures: the average absolute deviation between predictions and ground truth. Interpretable — if MAE = 5 and your target is house price in €1000s, you're off by €5,000 on average.

python

from sklearn.metrics import mean_absolute_error
import numpy as np

y_true = np.array([3, 5, 8, 10])
y_pred = np.array([2.5, 6, 7, 11])

mean_absolute_error(y_true, y_pred)   # → 0.875
np.abs(y_true - y_pred).mean()       # same
✓ MAE advantage

Robust to outliers — a single massive error doesn't dominate. Each error contributes linearly. Easier to interpret in the original units.

✗ MAE disadvantage

Non-differentiable at zero (subgradient needed). Doesn't penalize large errors more — treating a 1-unit error the same proportionally as a 100-unit error.

MAE vs MSE — the gradient insight
MAE gives constant gradient w.r.t. all errors regardless of size (sign only). MSE gives gradient proportional to error magnitude — large errors get larger gradient updates. If your data has genuine outliers you want to ignore: MAE. If large errors are systematically important (e.g., demand forecasting where stockouts are critical): MSE/RMSE.
08 MSE & RMSE
Regression
formulas MSE = (1/n) · Σ (yᵢ − ŷᵢ)²
RMSE = √MSE

MSE unit: target²    RMSE unit: same as target (interpretable)
Corresponds to Gaussian likelihood. Differentiable everywhere.

RMSE vs MAE: RMSE penalizes large errors quadratically — a 10-unit error contributes 100 to MSE vs 10 to MAE. This means RMSE is more sensitive to outliers and large mistakes. If you care more about avoiding large errors than many small ones, RMSE is the right choice.

python

from sklearn.metrics import mean_squared_error
import numpy as np

mse  = mean_squared_error(y_true, y_pred)
rmse = np.sqrt(mse)
# sklearn ≥1.0: mean_squared_error(y_true, y_pred, squared=False) → RMSE

# Huber loss: hybrid MAE+MSE, robust to outliers + differentiable at 0
def huber(y, yhat, delta=1.0):
    err = np.abs(y - yhat)
    return np.where(err <= delta,
                    0.5 * err**2,              # quadratic for small errors
                    delta * err - 0.5*delta**2  # linear for large errors
                   ).mean()
RMSE is not robust to scale
RMSE cannot be compared across datasets with different target scales. A model predicting house prices (error in €1000s) will have a larger RMSE than one predicting ages (error in years) — without implying it's worse. Always normalize or use relative metrics (MAPE, NRMSE) for cross-dataset comparison.
09 R² — Coefficient of Determination
Regression
formula R² = 1 − SS_res / SS_tot

SS_res = Σ(yᵢ − ŷᵢ)²   ← residual sum of squares (your model's error)
SS_tot = Σ(yᵢ − ȳ)²   ← total variance in y (baseline: predict mean)

R² = 0 → model performs no better than predicting the mean
R² = 1 → perfect predictions
R² < 0 → model is worse than predicting the mean (possible!)

What it measures: what fraction of the variance in y your model explains. R² = 0.85 means your model accounts for 85% of the variance in the target — the remaining 15% is unexplained noise or missing features.

python

from sklearn.metrics import r2_score

r2 = r2_score(y_true, y_pred)

# Manual
ss_res = ((y_true - y_pred)**2).sum()
ss_tot = ((y_true - y_true.mean())**2).sum()
r2 = 1 - ss_res / ss_tot

# Adjusted R² — penalizes adding useless features
n, p = len(y_true), X.shape[1]   # n samples, p features
r2_adj = 1 - (1 - r2) * (n - 1) / (n - p - 1)
R² can be negative — and is not always the right metric
R² can be negative when a model is worse than predicting the mean. This happens with overly biased models or out-of-distribution predictions. Also: R² always increases (or stays flat) as you add features — use Adjusted R² which penalizes added complexity. R² measures fit to training data — low training R² signals underfitting; high training R² + low test R² signals overfitting.
R² ≠ correlation²... usually
For simple linear regression with a single feature, R² = Pearson r². For multiple features or non-linear models, this equivalence breaks. Don't conflate them.
10 IoU — Intersection over Union (Jaccard)
Segmentation · Detection
formula IoU = |A ∩ B| / |A ∪ B|
= TP / (TP + FP + FN)

Range: [0, 1]. Also called Jaccard Index. TN not involved.
mIoU — mean IoU (multi-class segmentation) mIoU = (1/C) · Σ_c IoU_c   unweighted mean over C classes

What it measures: the spatial overlap between predicted mask and ground truth mask, penalizing both over-prediction (FP) and under-prediction (FN) in the denominator. The denominator is the union — everything either mask covers.

python

import numpy as np

def iou(pred: np.ndarray, target: np.ndarray, smooth: float = 1e-6) -> float:
    # pred, target: binary masks (H, W), dtype bool or 0/1
    intersection = (pred & target).sum()
    union         = (pred | target).sum()
    return (intersection + smooth) / (union + smooth)

# Multi-class mIoU with PyTorch
def miou(preds, targets, num_classes):
    ious = []
    for c in range(num_classes):
        pred_c   = (preds == c)
        target_c = (targets == c)
        inter    = (pred_c & target_c).sum().item()
        union    = (pred_c | target_c).sum().item()
        if union == 0:
            continue           # class absent in both pred and target
        ious.append(inter / union)
    return sum(ious) / len(ious)

# IoU threshold in object detection (COCO: IoU ≥ 0.5 = true positive)
# mAP@50: mean AP at IoU threshold 0.5
# mAP@50:95: mean AP averaged over IoU thresholds 0.5, 0.55, ..., 0.95
IoU vs Dice — the key difference in gradients
Both measure overlap, but their gradients behave differently during training. IoU's gradient is sparse — zero everywhere except at boundary pixels. Dice's gradient is smoother. In practice, Dice is preferred as a training loss; IoU is preferred as an evaluation metric (because it's more interpretable — an IoU of 0.7 means 70% overlap). Many implementations use Dice loss during training but report IoU for validation.
11 Dice Coefficient (F1 for Masks)
Segmentation
formula Dice = 2|A ∩ B| / (|A| + |B|)
= 2·TP / (2·TP + FP + FN)

relationship to IoU Dice = 2·IoU / (1 + IoU)   ← monotone transform: Dice ≥ IoU always
IoU = Dice / (2 − Dice)
Dice = F1 score at the pixel level. Range: [0, 1].

Why Dice ≥ IoU always: The denominator of Dice is |A| + |B| = 2·TP + FP + FN, while IoU's denominator is TP + FP + FN (the union). Since 2·TP > TP always (for TP > 0), the Dice denominator is smaller, the fraction is larger.

Dice as a loss function: Unlike IoU, Dice is differentiable and can be used directly as a training loss (with soft/probabilistic predictions replacing binary masks).

python

import torch

def dice_coefficient(pred, target, smooth=1e-6):
    # pred, target: binary masks, shape (B, H, W)
    intersection = (pred * target).sum(dim=(1,2))
    dice = (2 * intersection + smooth) / (
           pred.sum(dim=(1,2)) + target.sum(dim=(1,2)) + smooth
    )
    return dice.mean()

def dice_loss(pred_logits, target, smooth=1e-6):
    # Soft Dice loss — differentiable, works on probabilities
    pred  = torch.sigmoid(pred_logits)
    inter = (pred * target).sum(dim=(1,2))
    denom = pred.sum(dim=(1,2)) + target.sum(dim=(1,2))
    return 1 - ((2 * inter + smooth) / (denom + smooth)).mean()

# In practice: combine BCE + Dice for stability
loss = bce_loss(logits, target) + dice_loss(logits, target)

the smooth (ε) term
Adding a small constant ε (1e-6) to numerator and denominator prevents division by zero when both masks are empty and stabilizes gradients when predictions are all-zero at training start. It slightly biases the metric toward 1.0 for empty masks — be aware of this when evaluating rare classes.
12 Decision Guide — When to Use What
ScenarioPrimary metricWhy / notes
Balanced binary classificationAccuracy, F1Classes roughly equal — accuracy is interpretable. F1 for safety.
Imbalanced binary (rare positives)AUC-PR, F1AUC-ROC is too optimistic. F1 at your operating threshold.
Ranking / retrievalAUC-ROC, MAPThreshold-independent. You care about ordering, not a binary decision.
Cancer / fraud detection (FN costly)Recall, F2Missing a positive is much worse than a false alarm.
Spam filter (FP costly)Precision, F0.5Blocking legitimate email is worse than missing spam.

the question behind every metric choice
Before picking a metric, answer: "What is the cost of a false positive vs a false negative in my problem?" If FN costs more → optimize recall. If FP costs more → optimize precision. If equal → F1. If you don't know the relative costs → AUC-ROC or AUC-PR to defer the threshold decision to deployment time.
13 Before the Interview — Checklist
  • Can derive Accuracy, Precision, Recall, F1 from TP/TN/FP/FN alone, without looking them up
  • Knows why F1 uses harmonic mean: penalizes imbalance between precision and recall
  • Can explain why 99% accuracy is meaningless on a 99%-negative dataset
  • Knows AUC-ROC's probabilistic interpretation: P(score(pos) > score(neg))
  • Knows when AUC-ROC lies: under heavy class imbalance → use AUC-PR instead
  • Can explain the precision-recall tradeoff: raising threshold increases P, decreases R
  • Knows Fβ: β > 1 weights recall, β < 1 weights precision. Can give a use case for each.
  • MAE vs MSE: MAE robust to outliers (L1 loss), MSE penalizes large errors more (L2 loss)
  • Can explain R² = 1 − SS_res/SS_tot and why R² can be negative
  • Knows Adjusted R²: penalizes adding useless features that inflate raw R²
  • IoU = TP / (TP + FP + FN) — denominator is the union, TN never appears
  • Dice = 2·TP / (2·TP + FP + FN) — always ≥ IoU; Dice = 2·IoU/(1+IoU)
  • Dice as training loss (differentiable), IoU as evaluation metric (interpretable) — knows the difference
  • Knows the ε smoothing term in Dice/IoU: prevents div-by-zero, slightly biases toward 1 on empty masks
  • Can explain macro vs micro vs weighted F1 averaging and when each is appropriate
  • MCC includes all four confusion matrix cells — more balanced than F1 for imbalanced data
