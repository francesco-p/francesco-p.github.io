From pixels to tokens:
what changed, what didn't

You already know the fundamentals — tensors, gradients, attention, residual connections. LLMs are built from the same primitives, but the domain shift is deeper than it looks. Discrete inputs, autoregressive generation, next-token prediction as the universal objective, and training at a scale where the data itself becomes the model.

01 The mental model shift

In CV, your model sees a 2D spatial grid and predicts a label or a dense map. The spatial structure is the core inductive bias — convolutions exploit it, ViT attends over it. Language has no such grid. Instead it has sequence: a 1D stream of discrete symbols where order encodes meaning and the relationship between distant tokens matters as much as adjacent ones.

Computer Vision

Input: pixel grid (H×W×C) — continuous, fixed size
Output: class label, bounding box, segmentation mask
Structure: spatial locality — nearby pixels are related
Prediction: one-shot (or per-pixel) from full input

Language Models

Input: token sequence (T) — discrete, variable length
Output: distribution over next token (vocabulary size V)
Structure: sequential dependency — any token depends on all prior
Prediction: autoregressive — one token at a time, left to right

the one sentence that explains everything
An LLM is a function that takes a sequence of tokens and outputs a probability distribution over what the next token should be — and that single objective, applied at internet scale, produces a model that can reason, translate, code, and generalize.

Think of it like this: if you trained a ViT to predict the next patch given all previous patches in a rasterized image scan, you'd have something structurally identical to a GPT. That's actually how image GPT (iGPT) works. The architecture is the same — the domain is different.

02 Tokens & vocabulary

Images have pixels — continuous real-valued intensities, differentiable end to end. Language has tokens — discrete integer IDs from a fixed vocabulary. This discreteness is fundamental: you cannot backprop through a token index. Everything after tokenization is continuous, but the tokenizer itself is frozen during training.

CV analogy

Tokenization ≈ vector quantization (VQ-VAE). Continuous inputs are discretized into a finite codebook. After quantization, you work with codebook indices — just like token IDs.

LLM reality

Byte-Pair Encoding (BPE) merges frequent character pairs into subword units. "unbelievable" → ["un", "believ", "able"]. Vocabulary size V ≈ 32k–100k. OOV words are handled by falling back to bytes.

Byte-Pair Encoding (BPE)

01
Start with characters
Initialize vocabulary with all individual bytes/characters. Every word is a sequence of these base tokens.
02
Count pair frequencies
Find the most frequent adjacent pair across the entire corpus. e.g., ("e", "r") appears 12,000 times.
03
Merge the top pair
Create a new token "er" and replace all occurrences. Add merge rule to the rulebook.
04
Repeat until vocabulary full
Iterate until vocabulary reaches target size (e.g., 50,257 for GPT-2). Common words become single tokens; rare words are split into subwords.
why tokenization matters
Tokenization is a silent hyperparameter. Poor tokenization makes arithmetic hard (numbers split weirdly: "123" → ["12","3"]), hurts low-resource languages (Chinese characters may each be one token while English words average ~0.75 tokens), and inflates context length. GPT-4's cl100k tokenizer is dramatically better than GPT-2's for code and multilingual text.

03 The core objective: next-token prediction

Everything in LLM pretraining reduces to one loss: given a sequence of tokens, predict the next one. This is called causal language modeling (CLM) or autoregressive language modeling.

causal language modeling loss \(\mathcal{L} = -\dfrac{1}{T}\sum_{t=1}^{T}\log P(x_t\mid x_1,\ldots,x_{t-1})\)

= average negative log-likelihood of each token given all prior tokens
= cross-entropy between one-hot true token and softmax output

\(\text{Perplexity} = \exp(\mathcal{L})\) ← lower is better; 1 = perfect, ∞ = random
CV equivalent

Masked autoencoder (MAE): predict masked patches from visible patches. BERT-style: bidirectional. But language models are strictly causal — no peeking at future tokens.

LLM reality

Every forward pass produces T predictions simultaneously (one per position). The causal mask ensures position t only attends to positions ≤ t. Loss averaged over all positions = massive label efficiency.

why this objective is so powerful
Every text document is simultaneously a supervised dataset — each token is a label for the preceding context. A 1 trillion token corpus gives ~1 trillion training examples, all self-labeled. Predicting the next word forces the model to compress world knowledge, grammar, reasoning, and common sense into its weights — because all of these help predict text better.

04 Token embeddings

Discrete token IDs can't be fed directly into a neural net — they have no geometric meaning. The embedding table maps each integer ID to a dense learned vector of dimension d_model.

CV analogy

The first conv layer in a CNN transforms raw RGB pixels into learned feature maps. The embedding table does the same for tokens — it's a learned lookup table, equivalent to a linear layer with a one-hot input.

LLM reality

nn.Embedding(V, d_model) — a matrix of shape (V, d_model). Row i is the embedding for token i. Weights are learned. Often tied with the final output projection (weight tying).

pytorch
import torch
import torch.nn as nn

V       = 50257   # vocabulary size (GPT-2)
d_model = 768     # embedding / model dimension
T       = 1024    # context length (sequence length)

token_emb = nn.Embedding(V, d_model)

tokens = torch.randint(0, V, (batch := 4, T))  # (B, T) integer IDs
x = token_emb(tokens)                             # (B, T, d_model)

# Weight tying: share embedding & output weights (saves params, improves perf)
lm_head = nn.Linear(d_model, V, bias=False)
lm_head.weight = token_emb.weight                   # tied

05 Positional encoding

Self-attention is permutation-invariant — shuffling the token sequence would produce the same attention weights. We need to inject position information explicitly.

Learned absolute (GPT-2)

A second embedding table of shape (T_max, d_model). Position 0–1023 each get a learned vector. Simple and effective. Fails to generalize beyond T_max.

Sinusoidal (original Transformer)

Fixed: PE(pos,2i) = sin(pos/10000^(2i/d)). No learned params. Smooth encoding. Can extrapolate but not well in practice.

RoPE — Rotary Position Embedding

Modern standard (LLaMA, GPT-NeoX). Encodes relative position by rotating Q and K vectors before dot product. Attention score depends only on relative offset → generalizes better to longer sequences.

ALiBi

Adds a linear penalty to attention logits based on distance. No positional vectors at all. Simple, extrapolates well. Used in MPT, BLOOM.

RoPE — why it works
For query q at position m and key k at position n, RoPE ensures: qₘᵀkₙ depends only on (m − n), not on absolute positions. This means the model learns position-relative patterns — "token 3 positions back" — rather than absolute-position-specific patterns. The result: much better length generalization. 
pytorch — learned positional
pos_emb = nn.Embedding(T, d_model)
positions = torch.arange(T, device=tokens.device)  # [0, 1, ..., T-1]
x = token_emb(tokens) + pos_emb(positions)          # (B, T, d_model)
# After this: x is the initial representation fed into transformer blocks

06 Causal self-attention — the core operation

In CV, each spatial location attends to all others (ViT). In a language model, position t can only attend to positions ≤ t — the causal mask prevents seeing the future. This is what makes generation possible: at inference, we generate left to right.

scaled dot-product attention (single head) \(Q = XW_Q,\quad K = XW_K,\quad V = XW_V\)
X: (B,T,d) · W_Q,W_K,W_V: (d,d_head) · Q,K,V: (B,T,d_head)

\(\text{scores} = QK^T\!/\sqrt{d_{\text{head}}}\) (B,T,T)
\(\text{scores} = \text{scores} + \text{causal\_mask}\) ← set future positions to −∞
\(\text{weights} = \operatorname{softmax}(\text{scores})\) (B,T,T) — rows sum to 1
\(\text{output} = \text{weights}\cdot V\) (B,T,d_head)
ViT attention

Bidirectional — every patch attends to every other patch. No mask. The [CLS] token aggregates all patches. Used for classification, not generation.

GPT attention

Causal (autoregressive) — position t attends only to ≤ t. Lower-triangular mask. No [CLS] token. The last position is used for next-token prediction.

pytorch — causal self-attention
import torch.nn.functional as F

class CausalSelfAttention(nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.n_heads = n_heads
        self.d_head  = d_model // n_heads
        self.qkv  = nn.Linear(d_model, 3 * d_model, bias=False)
        self.proj = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x):
        B, T, C = x.shape
        # Project to Q, K, V in one shot, then split
        q, k, v = self.qkv(x).chunk(3, dim=-1)

        # Reshape to (B, n_heads, T, d_head)
        def split_heads(t):
            return t.view(B, T, self.n_heads, self.d_head).transpose(1, 2)
        q, k, v = split_heads(q), split_heads(k), split_heads(v)

        # Flash attention (PyTorch 2.0+) — fused, memory-efficient
        out = F.scaled_dot_product_attention(q, k, v, is_causal=True)

        # Merge heads and project
        out = out.transpose(1, 2).contiguous().view(B, T, C)
        return self.proj(out)
Flash Attention — why it matters
Standard attention materializes the (B, H, T, T) score matrix in HBM (GPU DRAM) — O(T²) memory. FlashAttention fuses the softmax + matmul into a single CUDA kernel with tiled computation in fast SRAM. Same numerics, same output — but O(T) memory and 2–4× faster on long sequences. PyTorch 2.0's F.scaled_dot_product_attention uses it automatically.

07 Multi-head attention & variants

Running h attention heads in parallel lets the model attend to different representation subspaces simultaneously. Each head learns different relationships — syntax, coreference, semantics.

multi-head attention \(\text{head}_i = \operatorname{Attention}(QW_i^Q,\,KW_i^K,\,VW_i^V)\)
\(\operatorname{MHA}(X) = \operatorname{Concat}(\text{head}_1,\ldots,\text{head}_h)\cdot W^O\)
\(d_{\text{head}} = d_{\text{model}}/h\) · total params per layer \(\approx 4d_{\text{model}}^2\)
VariantK/V headsUsed inBenefit
MHA (standard)= Q headsGPT-2, original TransformerFull expressivity
MQA (Multi-Query)1 shared headPaLM, FalconMuch smaller KV cache → faster inference
GQA (Grouped-Query)G groups (G<H)LLaMA-2/3, MistralBalance: quality ≈ MHA, speed ≈ MQA
why GQA is the modern default
The KV cache (keys and values for all prior tokens) dominates memory during inference. With 32 Q heads and only 8 KV heads (GQA), the KV cache is 4× smaller. LLaMA-3 8B uses 8 KV heads / 32 Q heads — the quality gap vs full MHA is negligible at scale, but the inference speedup is substantial.

08 The Transformer block

A single transformer block = self-attention + feed-forward network (FFN), each wrapped in a residual connection and layer norm. The entire LLM is just this block stacked N times.

pytorch — transformer block
class TransformerBlock(nn.Module):
    def __init__(self, d_model, n_heads, ffn_mult=4, dropout=0.0):
        super().__init__()
        self.ln1  = nn.RMSNorm(d_model)       # modern: RMSNorm over LayerNorm
        self.attn = CausalSelfAttention(d_model, n_heads)
        self.ln2  = nn.RMSNorm(d_model)
        self.ffn  = nn.Sequential(
            nn.Linear(d_model, ffn_mult * d_model, bias=False),
            nn.GELU(),
            nn.Linear(ffn_mult * d_model, d_model, bias=False),
            nn.Dropout(dropout),
        )

    def forward(self, x):
        x = x + self.attn(self.ln1(x))    # pre-norm + residual
        x = x + self.ffn(self.ln2(x))     # pre-norm + residual
        return x
what does the FFN actually do?
The attention mechanism routes information between tokens. The FFN processes each token independently — it's a per-position transformation. Researchers have shown FFN layers act like soft key-value memories: the first linear layer selects which "facts" are relevant, the second writes the associated value into the residual stream. This is why FFN layers store most of the factual knowledge in LLMs.

09 Full GPT-style architecture

pytorch — minimal GPT
class GPT(nn.Module):
    def __init__(self, V, T, d_model, n_heads, n_layers, dropout=0.0):
        super().__init__()
        self.tok_emb = nn.Embedding(V, d_model)
        self.pos_emb = nn.Embedding(T, d_model)
        self.blocks  = nn.ModuleList([
            TransformerBlock(d_model, n_heads, dropout=dropout)
            for _ in range(n_layers)
        ])
        self.ln_f    = nn.RMSNorm(d_model)   # final norm before lm_head
        self.lm_head = nn.Linear(d_model, V, bias=False)
        self.lm_head.weight = self.tok_emb.weight  # weight tying

    def forward(self, idx, targets=None):
        B, T = idx.shape
        pos  = torch.arange(T, device=idx.device)
        x    = self.tok_emb(idx) + self.pos_emb(pos)
        for block in self.blocks:
            x = block(x)
        logits = self.lm_head(self.ln_f(x))  # (B, T, V)

        loss = None
        if targets is not None:
            loss = F.cross_entropy(
                logits.view(-1, V), targets.view(-1)
            )
        return logits, loss

10 KV cache — the inference trick

Autoregressive generation calls the model once per new token. Without caching, each forward pass recomputes keys and values for all prior tokens — \(O(T^2)\) total work for a sequence of length T. The KV cache eliminates this by storing the K and V tensors for every position already generated.

CV analogy

Like feature caching in a sliding-window detector — compute features for a region once and reuse them. In LLMs, every past token position is a "region" whose keys and values never change once generated.

LLM reality

Cache stores (K, V) tensors of shape (B, n_kv_heads, T_so_far, d_head) per layer. At step t+1 you only compute Q for the single new token, then attend over the full cached K and V. Per-step cost drops from \(O(T^2)\) to \(O(T)\).

Prefill vs Decode

Generation has two distinct phases with very different bottlenecks:

PhaseWhat happensBottleneck
PrefillProcess the entire prompt in one forward pass. Build the initial KV cache from scratch.Compute-bound (many tokens processed in parallel)
DecodeGenerate one token at a time. Each step reads the full KV cache + appends one new (K,V) pair.Memory-bandwidth-bound (reading the whole cache each step)
why decode is memory-bandwidth-bound
At each decode step you load the entire KV cache — potentially gigabytes — from HBM to compute just one token. GPU HBM bandwidth is ~3 TB/s; you're reading ~GB per step → only a few hundred tokens/sec on one GPU. This is why batching decode requests together matters: you pay the HBM read cost once across many requests in a batch.
KV cache memory formula \(\text{KV size} = 2 \times L \times H_{kv} \times d_h \times T \times B \times \text{bytes}\)
L: layers · H_kv: KV heads · d_h: head dim · T: seq len · B: batch · 2 for K and V

LLaMA-3 70B example: \(L{=}80,\;H_{kv}{=}8,\;d_h{=}128,\;T{=}4096,\;B{=}1\), fp16:
\(2\times80\times8\times128\times4096\times1\times2\,\text{B} \approx \mathbf{8\,\text{GB}}\)

KV cache reduction techniques

GQA / MQA

Grouped-Query Attention uses fewer KV heads than Q heads (e.g., 8 vs 32 in LLaMA-3). Cache shrinks by the ratio. MQA goes further: a single shared KV head — maximum memory savings, small quality cost.

Paged Attention (vLLM)

KV cache is fragmented as sequences grow to different lengths. vLLM uses non-contiguous "pages" (like virtual memory) so GPU memory is never wasted on padding or pre-allocation. Enables much larger batch sizes.

Flash-Decoding

Parallelizes the attention over the sequence length dimension during decode (standard Flash Attention parallelizes over batch/heads). Enables fast single-sequence decode on many GPU cores simultaneously.

KV quantization

Quantize cached K and V to INT8 or INT4 (vs fp16). Halves or quarters the memory with small quality loss. Used in production at most major inference providers. Q stays in fp16 for quality.

the memory wall: why this limits context length in practice
Longer context = larger KV cache. With 8 GB for 4K tokens, an 128K context window needs ~256 GB just for the KV cache of one sequence — more than most single-node GPU memory. This is why sliding-window attention (Mistral), sparse attention, and retrieval-augmented generation all exist: they trade exact long-range attention for memory efficiency.

11 Data & the pretraining corpus

Pretraining data is the model's world. The quality, diversity, and scale of the corpus determines what the model knows, its biases, and its capability ceiling. At this scale, data engineering is a primary research problem.

01
Raw crawl
CommonCrawl: petabytes of raw web text. Includes boilerplate, spam, duplicates, low-quality content. Starting point for most open models.
02
Quality filtering
Perplexity filtering (discard text a reference LM finds surprising), classifier-based filtering (train a model to score quality vs Wikipedia), length filters, language ID.
03
Deduplication
MinHash LSH to find near-duplicate documents. Exact-match dedup at substring level. Critical: duplicated data causes memorization and degrades generalization.
04
Domain mixing
Mix web text, books, code, Wikipedia, scientific papers with carefully tuned weights. Code dramatically improves reasoning. More math data → better math. Ratios are major hyperparameters.
05
Tokenize & pack
Tokenize all documents, concatenate into one stream, chunk into fixed-length sequences (e.g., T=2048). A sequence may span multiple documents separated by <|endoftext|>.

12 Pretraining at scale

The training loop looks deceptively similar to any other supervised setup, but the engineering to make it work at 1000+ GPUs without divergence is where the real complexity lives. 

pytorch — pretraining loop skeleton
from torch.distributed import init_process_group
from torch.nn.parallel import DistributedDataParallel as DDP

# Distributed setup (one process per GPU)
init_process_group(backend="nccl")
model = GPT(...).cuda()
model = DDP(model)                      # sync gradients across GPUs

optimizer = torch.optim.AdamW(
    model.parameters(), lr=3e-4,
    betas=(0.9, 0.95), weight_decay=0.1  # common LLM hyperparams
)
scaler = torch.amp.GradScaler()          # mixed precision

for step, (x, y) in enumerate(dataloader):
    optimizer.zero_grad(set_to_none=True)  # set_to_none faster than zero

    with torch.amp.autocast(device_type="cuda", dtype=torch.bfloat16):
        _, loss = model(x, targets=y)

    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(optimizer); scaler.update()
    lr_schedule(optimizer, step)          # warmup + cosine decay
Data parallelism (DDP)

Each GPU holds a full model copy. Each sees a different data batch. Gradients are averaged across all GPUs after backward. Works up to ~100s of GPUs.

Tensor parallelism (TP)

Split individual weight matrices across GPUs. Each GPU holds a shard of each layer. Required when model doesn't fit on one GPU. Megatron-LM style.

Pipeline parallelism (PP)

Different layers on different GPUs. GPU 0 holds layers 0–23, GPU 1 holds 24–47, etc. Micro-batching hides the bubble overhead. Used for 100B+ models.

ZeRO (DeepSpeed)

Shards optimizer states, gradients, and params across GPUs (Stage 1/2/3). Eliminates memory redundancy of naive DDP. Enables training large models on modest hardware.

13 Scaling laws

Kaplan et al. (2020) and Hoffmann et al. (Chinchilla, 2022) found that LLM loss follows smooth power laws in model size N, dataset size D, and compute C — predictable enough to plan entire training runs in advance.

Chinchilla scaling law (optimal compute allocation) L(N, D) ≈ A/Nᵅ + B/Dᵝ + L∞

Optimal: N ∝ C^0.5   D ∝ C^0.5
→ tokens = ~20× parameters for compute-optimal training
e.g., 70B model → train on ~1.4T tokens (LLaMA-2 did exactly this)
the Chinchilla insight
GPT-3 (175B) was undertrained — only ~300B tokens. A 70B model trained on 1.4T tokens (same compute) outperforms it on most benchmarks. The field has shifted: smaller, longer-trained models beat large undertrained ones at equal compute. LLaMA-3 8B trained on 15T tokens — 75× more than Chinchilla-optimal — because inference cost matters, not just training cost.

14 Training instabilities at scale

Loss spikes

Sudden large increases in loss mid-training. Often caused by a batch of "hard" or corrupted data. Fix: data quality filtering, gradient clipping, restart from checkpoint before spike.

Gradient explosion

Gradients blow up, NaN propagates. Fix: gradient norm clipping (max_norm=1.0), lower learning rate, better initialization, smaller microbatch size.

Attention entropy collapse

Attention weights become nearly one-hot ("attention sink"). Model focuses on one token, loses context. Fix: QK-norm, temperature scaling, attention regularization.

Rank collapse in layers

Weight matrices lose rank — representations become low-dimensional. Fix: weight decay, proper initialization (μP, Kaiming), avoid very high learning rates.

15 Supervised fine-tuning (SFT)

After pretraining, the model is a next-token predictor on raw text. It can complete documents but doesn't follow instructions. SFT teaches it to follow the chat format: given a user message, generate a helpful assistant response.

chat format — typical template <|system|> You are a helpful assistant.
<|user|> Explain gradient descent in one paragraph.
<|assistant|> Gradient descent is an optimization...
← loss computed only on assistant tokens (not user / system)
SFT data is tiny but impactful
A typical SFT dataset is 10k–1M examples — orders of magnitude smaller than pretraining. Quality matters far more than quantity. The LIMA paper showed 1,000 carefully curated examples can outperform 50,000 noisy ones. SFT doesn't teach new knowledge — it teaches the model to surface knowledge it already has in the desired format and style. 
pytorch — SFT loss (mask non-assistant tokens)
# Only compute loss on assistant response tokens
labels = input_ids.clone()
labels[attention_mask == 0] = -100        # padding
labels[is_user_or_system_token] = -100    # prompt tokens — no loss here

logits, _ = model(input_ids)
loss = F.cross_entropy(
    logits[:, :-1, :].reshape(-1, V),
    labels[:, 1:].reshape(-1),
    ignore_index=-100                       # ignore masked positions
)

16 RLHF — Reinforcement Learning from Human Feedback

SFT teaches format. RLHF teaches preference — helpful, harmless, honest responses over technically correct but unhelpful or harmful ones. It's how ChatGPT, Claude, and Gemini become useful assistants rather than raw text completers.

01
Collect comparisons
For each prompt, sample 2–8 completions from the SFT model. Human annotators rank them by quality (helpfulness, safety, accuracy). Creates a preference dataset of (prompt, chosen, rejected) triples.
02
Train reward model (RM)
Fine-tune a LM (often the SFT model + a scalar head) to predict human preference. Trained with Bradley-Terry loss: reward(chosen) > reward(rejected). The RM is a proxy for human judgment.
03
RL fine-tuning (PPO)
The SFT model is the policy. For each prompt, generate a completion → reward model scores it → PPO updates the policy to maximize reward. A KL penalty keeps the policy close to the SFT model (prevents reward hacking).
RLHF objective max E[R(x, y)] − β · KL(π_θ(y|x) ‖ π_SFT(y|x))
← maximize reward while staying close to the SFT model
β controls alignment tax — larger β = more conservative policy
reward hacking
The policy finds completions that maximize the RM score without actually being good. e.g., long, verbose answers score higher if the RM was trained on length-biased data. The KL term and careful RM training mitigate this. It's an ongoing research problem.

17 DPO — Direct Preference Optimization

DPO (Rafailov et al., 2023) is an elegant alternative to RLHF that skips the reward model entirely. It reparameterizes the RL problem to optimize preferences directly on the LM — same outcome, much simpler training.

DPO loss L_DPO = −E[log σ(β · log(π_θ(y_w|x)/π_ref(y_w|x)) − β · log(π_θ(y_l|x)/π_ref(y_l|x)))]

y_w = preferred (won) · y_l = dispreferred (lost) · π_ref = frozen SFT model
Intuition: increase log-prob of chosen, decrease log-prob of rejected, relative to reference. 
pytorch — DPO loss (simplified)
def dpo_loss(policy, reference, chosen_ids, rejected_ids, beta=0.1):
    with torch.no_grad():
        ref_chosen  = log_prob(reference, chosen_ids)
        ref_rejected = log_prob(reference, rejected_ids)

    pol_chosen   = log_prob(policy, chosen_ids)
    pol_rejected = log_prob(policy, rejected_ids)

    # Log-ratio difference (relative to reference)
    delta = (pol_chosen - ref_chosen) - (pol_rejected - ref_rejected)
    loss  = -F.logsigmoid(beta * delta).mean()
    return loss
DPO vs RLHF in practice
DPO is simpler (no RM training, no PPO rollouts), more stable, and often competitive. RLHF with online sampling is more powerful when you can generate many diverse completions — it sees novel responses not in the dataset. DPO is offline — it only sees (chosen, rejected) pairs. Most open models today use DPO or variants (KTO, IPO, SimPO).

18 Parameter-Efficient Fine-Tuning (PEFT) & LoRA

Fine-tuning a 70B model updates 70B parameters — unaffordable on consumer hardware. PEFT methods update a tiny fraction of parameters while keeping the base model frozen.

LoRA — Low-Rank Adaptation W' = W₀ + ΔW = W₀ + BA
W₀: d×k frozen · B: d×r · A: r×k · r ≪ min(d,k)

Trainable params: r·(d + k) vs d·k for full fine-tune
At r=8, d=k=4096: 65,536 params vs 16,777,216 — 256× fewer
CV analogy

Like adapter layers in domain adaptation — freeze the backbone, add small trainable modules between layers. BitFit (train only biases) is even more extreme. Same philosophy: minimal params, maximum leverage.

LoRA specifics

Applied to Q, K, V, and output projection matrices. B initialized to zero (so ΔW=0 at start, no disturbance). A initialized with Gaussian. At inference, merge W₀ + BA — no latency overhead.

pytorch — LoRA layer
class LoRALinear(nn.Module):
    def __init__(self, linear: nn.Linear, rank=8, alpha=16):
        super().__init__()
        d, k = linear.weight.shape
        self.linear = linear
        self.linear.weight.requires_grad_(False)  # freeze base
        self.A = nn.Parameter(torch.randn(rank, k) * 0.01)
        self.B = nn.Parameter(torch.zeros(d, rank))
        self.scale = alpha / rank                     # α/r scaling factor

    def forward(self, x):
        base = self.linear(x)
        lora = (x @ self.A.T) @ self.B.T * self.scale  # low-rank delta
        return base + lora

    def merge(self):                               # merge for zero-latency inference
        self.linear.weight.data += (self.B @ self.A) * self.scale

19 Decoding strategies

At inference, the model outputs a probability distribution over V tokens. How you sample from that distribution determines whether you get creative, deterministic, diverse, or incoherent text.

StrategyHowWhen to use
Greedyargmax each stepFast, deterministic. Repetitive. Never good for creative tasks.
Beam searchKeep top-k sequences at each stepBetter for translation/summarization (short, factual). Bad for open-ended generation.
Temperaturelogits /= T before softmaxT<1: sharper (more certain). T>1: flatter (more random). T=0 → greedy.
Top-k samplingSample from top k tokens onlyAvoids very unlikely tokens. k=50 is common. Rigid — k doesn't adapt to distribution shape.
Top-p (nucleus)Sample from smallest set summing to pAdaptive cutoff. p=0.9 is standard for chat. Better than top-k.
Min-pDiscard tokens below p×p_maxNewer, often better than top-p. Scales threshold to the max prob token.
 
pytorch — generation loop
@torch.no_grad()
def generate(model, idx, max_new_tokens, temperature=1.0, top_p=0.9):
    for _ in range(max_new_tokens):
        logits, _ = model(idx)                # (B, T, V)
        logits = logits[:, -1, :] / temperature  # last position only

        probs = F.softmax(logits, dim=-1)

        # Top-p (nucleus) sampling
        sorted_p, sorted_idx = torch.sort(probs, descending=True)
        cum_p = torch.cumsum(sorted_p, dim=-1)
        mask  = (cum_p - sorted_p) > top_p    # remove tokens beyond nucleus
        sorted_p[mask] = 0
        sorted_p /= sorted_p.sum()            # renormalize
        probs.scatter_(-1, sorted_idx, sorted_p)

        next_token = torch.multinomial(probs, num_samples=1)
        idx = torch.cat([idx, next_token], dim=1)

    return idx

20 Inference optimization

Quantization (INT8/INT4)

Represent weights in lower precision. GPTQ, AWQ, bitsandbytes. 4-bit quantization cuts memory ~4× with <1% quality loss on most tasks. Enables 70B on a single A100.

Speculative decoding

A small draft model proposes K tokens; the large model verifies all K in one parallel forward pass. If accepted, K tokens are generated at the cost of ~1 large model step. ~2–3× speedup.

Continuous batching

Don't wait for all sequences in a batch to finish before starting new ones. Insert new sequences as slots free. Used by vLLM, TGI. Dramatically improves GPU utilization in serving.

Paged attention (vLLM)

Treats KV cache like OS virtual memory — pages allocated on demand, no fragmentation. Enables much larger effective batch sizes and longer sequences at serving time.

21 Context window & prompting

CV analogy

Context window ≈ input image resolution. Larger context = more information the model can use — but cost scales quadratically (O(T²) attention). Like high-res vs low-res inference.

LLM reality

Everything in the context window is "visible" to the model simultaneously. Few-shot examples, system prompt, conversation history, retrieved documents — all compete for the same T tokens.

the "lost in the middle" problem
Models attend more strongly to tokens at the beginning and end of the context. Information in the middle is less reliably used — demonstrated empirically across GPT-4, Claude, and open models. For RAG systems: put the most important retrieved chunks at the start or end of the context.
chain-of-thought (CoT) prompting
Prompting the model to "think step by step" before giving a final answer dramatically improves accuracy on reasoning tasks. Why: each intermediate step becomes a token in context, giving the model a "scratch pad". The model doesn't need to compute 7 reasoning steps in one attention pass — it can externalize computation into text.

pytorch — SwiGLU FFN (LLaMA-style)
class SwiGLUFFN(nn.Module):
    def __init__(self, d_model, hidden):
        super().__init__()
        self.gate = nn.Linear(d_model, hidden, bias=False)
        self.up   = nn.Linear(d_model, hidden, bias=False)
        self.down = nn.Linear(hidden,  d_model, bias=False)

    def forward(self, x):
        # SwiGLU: gate(x) * swish(up(x)) then project down
        return self.down(F.silu(self.gate(x)) * self.up(x))

23 Mixture of Experts (MoE)

Instead of one dense FFN per transformer block, MoE places N expert FFNs in parallel and a lightweight router that selects which top-k experts process each token. The key insight: parameters scale without proportionally scaling compute. GPT-4 is estimated to be a ~1.8T parameter MoE; each forward pass activates roughly ~220B parameters per token.

MoE FFN layer — forward pass \(g_i(x) = \operatorname{Softmax}(W_{\text{router}}\cdot x)_i\) ← router score for expert i
\(\mathcal{T}(x) = \operatorname{TopK}(\{g_i(x)\}_{i=1}^N,\;k)\) ← indices of selected experts
\(\operatorname{MoE}(x) = \sum_{i\in\mathcal{T}(x)} g_i(x)\cdot\operatorname{FFN}_i(x)\) ← gated sum of active experts

\(\text{active params per token} = \tfrac{k}{N}\times\text{total FFN params}\)
Mixtral 8×7B: N=8, k=2 → 46.7B total, ~12.9B active · DeepSeek-V3: N=256, k=8

Router mechanics

The router is a small linear layer \(W_r \in \mathbb{R}^{d \times N}\) trained jointly with the rest of the model. It maps each token's hidden state to N logits, takes softmax, then selects the top-k. The routing is per token — different tokens in the same sequence can go to different experts. This is what allows specialization: some experts learn syntax, others factual recall, others code.

Load balancing loss

Without it, the router collapses into "rich get richer" mode — always picking the same 1–2 experts (routing collapse). The auxiliary loss penalizes imbalanced expert utilization across a batch: \(\mathcal{L}_{\text{aux}} = \alpha\sum_i f_i P_i\), where \(f_i\) is fraction of tokens routed to expert i and \(P_i\) is average router probability for expert i.

Expert capacity & token dropping

In batch-parallel training, each expert has a capacity C = (tokens_per_batch / N) × capacity_factor. If more tokens are routed to an expert than its capacity allows, overflow tokens are dropped (or passed through unchanged via the residual). This means MoE models need large batches to work well.

Expert parallelism

In distributed training, different experts live on different GPUs. An all-to-all communication collective sends each token to its assigned expert's GPU, runs the FFN there, then gathers results back. This all-to-all is the main communication overhead — scales with N × k × batch_size × d_model.

Expert specialization

Empirically, experts do specialize: interpretability studies find some experts activate mainly on code tokens, others on foreign languages, others on numbers. This emergent specialization is one reason MoE quality matches dense models — it's implicit conditional computation.

MoE trade-offs — the full picture
Win (training): for a fixed compute budget (FLOPs), MoE can have far more total parameters than a dense model. More parameters → better quality. Chinchilla scaling laws were derived for dense models — MoE models are more parameter-efficient for the same FLOPs.

Win (inference throughput): lower FLOPs per token → faster throughput at the same hardware if you can keep all experts in memory.

Cost (memory): all N experts must reside in GPU memory even though only k are active per step. A 1.8T MoE requires ~900 GB of memory at fp16 — far beyond a single node. Experts are typically sharded across many GPUs, requiring expensive all-to-all communication at each layer.

Cost (latency): for a single request, FLOPs are low but the all-to-all communication adds latency. Dense models often win on single-request latency; MoE wins on throughput at high batch sizes.
DeepSeek-V3 / R1 innovations
DeepSeek-V3 uses 256 experts with top-8 routing and a "shared expert" (always active, for common knowledge) plus 256 "routed experts". They add a no-auxiliary-loss load balancing strategy — instead of a gradient-based auxiliary loss, they use a bias term added to router logits that's updated heuristically to maintain balance. This avoids the quality–balance trade-off inherent in the auxiliary loss approach.

24 Key interview questions

why can't you just use a bidirectional transformer (like BERT) for generation?
BERT sees the full sequence — future tokens influence past representations. During generation you don't have future tokens yet. Autoregressive generation requires causal masking: each position can only depend on what came before. BERT-style models are discriminative (classify, embed) not generative. You'd need to mask randomly during training and sample sequentially — that's masked language modeling, not CLM.
what does a transformer layer actually do to a token's representation?
Think of the residual stream as a shared message-passing bus. Attention reads from other tokens' representations and writes relevant information into the stream. The FFN reads from the stream and writes stored "facts" or transformations back in. Layer norm stabilizes the magnitude. After N layers, each token's vector is a function of its entire causal context — it has "gathered" relevant context from all prior positions.
how is LLM training different from training a ResNet at scale?
Same fundamentals (SGD, cross-entropy), but: (1) sequence data requires causal masking, not random crop; (2) the "dataset" never fits in RAM — streamed from disk; (3) distributed training is unavoidable (no single GPU holds the model); (4) instabilities are common at scale and require checkpoint restarts; (5) the loss plateau matters more than the loss value — perplexity improvements translate unpredictably to downstream capability; (6) data curation is a primary research problem, not preprocessing.
why does in-context learning work?
ICL is not gradient-based learning — no weights update. The model "implements" a learning algorithm in the forward pass by attending to the examples in context and adapting its output accordingly. Mechanistically, attention heads can implement something like gradient descent by reading and using key-value pairs from the context. It emerges from pretraining on diverse data with recurring patterns of examples followed by conclusions.
what causes hallucination?
Multiple interacting causes: (1) the model learned to generate plausible-sounding text, not verified facts; (2) factual knowledge is distributed across FFN layers — retrieval is probabilistic, not exact; (3) training on conflicting or outdated data; (4) the generation process is autoregressive — each token conditions on all prior generated tokens, so a wrong token compounds; (5) RLHF can amplify confident, fluent-sounding responses even when incorrect. Mitigation: RAG (ground in retrieved documents), tool use, chain-of-thought for verification.

  • Understands SFT: what it teaches (format, not knowledge), why prompt masking matters

The frame that ties it all together: an LLM is a lossy compression of the internet into a differentiable function. Pretraining compresses world knowledge. SFT teaches the interface. RLHF aligns the objective. Prompting is the query language. Your CV background — attention, residuals, distributed training, scale — is almost directly transferable. The shift is from spatial to sequential, from continuous to discrete, and from predicting labels to predicting probability distributions over a vocabulary. Everything else is engineering at scale.
